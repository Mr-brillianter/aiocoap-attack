# 故障排除指南

## 问题 1: 客户端收到了第一个响应（不应该）

### 症状

```
2026-01-03 13:19:59,170 - MITM-Demo - INFO - 📥 [响应 #1] 收到响应
2026-01-03 13:19:59,170 - MITM-Demo - INFO -    Token: 8e2a
2026-01-03 13:19:59,170 - MITM-Demo - INFO -    状态: 4.04 Not Found
```

客户端收到了第一个响应，但按照攻击逻辑，第一个响应应该被攻击者缓存，不应该转发给客户端。

### 根本原因

**aiocoap 库的行为**：当你使用 `aiocoap` 发送请求时，它会：

1. 解析 URI（例如 `coap://127.0.0.1:5684/time`）
2. 创建到目标地址的连接
3. 发送请求
4. 等待响应

**问题**：即使你指定了代理地址（5684），aiocoap 也会直接连接到该地址，但是：
- 如果服务器在 5683 端口
- 客户端连接到 5684 端口（攻击者）
- 攻击者转发请求到 5683 端口（服务器）
- **服务器会直接响应给客户端的原始地址，而不是通过攻击者**

这是因为 UDP 是无连接的，服务器看到的源地址是客户端的地址，而不是攻击者的地址。

### 解决方案

有几种方法可以解决这个问题：

#### 方案 1: 使用原始 Socket（推荐）

不使用 aiocoap 库，直接使用 socket 发送 CoAP 请求。

**优点**：
- 完全控制数据包
- 可以确保通过攻击者

**缺点**：
- 需要手动构造 CoAP 消息
- 需要手动处理重传等逻辑

**示例**：参见 `test_attacker_simple.py`

#### 方案 2: 修改攻击者实现 NAT

让攻击者修改数据包的源地址，使服务器认为请求来自攻击者。

**实现**：
```python
# 攻击者转发请求时，修改源地址
# 这需要使用 Scapy 或其他底层网络库
```

**优点**：
- 客户端可以继续使用 aiocoap
- 更接近真实的 MITM 攻击

**缺点**：
- 需要 Scapy
- 需要管理员权限
- 实现复杂

#### 方案 3: 使用 iptables/netfilter（Linux）

在 Linux 上，可以使用 iptables 重定向流量。

```bash
# 重定向所有到 5683 的流量到 5684
iptables -t nat -A OUTPUT -p udp --dport 5683 -j REDIRECT --to-port 5684
```

**优点**：
- 透明代理
- 客户端无需修改

**缺点**：
- 仅限 Linux
- 需要 root 权限

### 当前实现的限制

当前的 MITM 实现有一个限制：

**它只能拦截发送到攻击者端口（5684）的请求，但无法阻止服务器直接响应给客户端。**

这是因为：
1. 客户端发送请求到攻击者（5684）✅
2. 攻击者转发请求到服务器（5683）✅
3. 服务器看到的源地址是客户端的地址（不是攻击者）❌
4. 服务器直接响应给客户端❌

### 推荐的测试方法

使用 `test_attacker_simple.py` 进行测试：

```bash
# 终端 1: 启动服务器
uv run python server.py

# 终端 2: 启动攻击者
uv run python attacker.py

# 终端 3: 运行简单测试
uv run python test_attacker_simple.py
```

这个测试使用原始 socket，可以确保流量通过攻击者。

## 问题 2: logger.info() 错误

### 症状

```
TypeError: Logger.info() missing 1 required positional argument: 'msg'
```

### 原因

`logger.info()` 需要一个消息参数，不能为空。

### 解决方案

```python
# 错误
logger.info()

# 正确
logger.info("")
```

已修复。

## 问题 3: 4.04 Not Found

### 症状

```
状态: 4.04 Not Found
```

### 原因

服务器没有 `/time` 路径。

### 解决方案

已在 `server.py` 中添加 `TimeResource` 类和 `/time` 路径。

## 总结

当前的 MITM 实现是一个**教育演示**，展示了中间人攻击的概念。

**限制**：
- 无法完全拦截 UDP 响应（服务器会直接响应给客户端）
- 需要客户端主动连接到攻击者端口

**适用场景**：
- 理解 MITM 攻击原理
- 学习 CoAP 协议
- 演示 Token 重用漏洞

**如果需要完整的 MITM 实现**：
- 使用 Scapy 修改数据包源地址
- 使用 iptables 重定向流量
- 或者使用原始 socket 手动构造请求
